# История изменений

Все заметные изменения этого проекта документируются в этом файле.

Формат основан на [Keep a Changelog](https://keepachangelog.com/ru/1.1.0/),
проект использует [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Невыпущено]

## [0.4.5] - 2026-06-21

### Добавлено

- Установщик `install-eval-tools`, который доставляет оснастку модельных
  прогонов из пакета навыка `ai-apm-setup` в целевой проект по манифесту
  `eval-tools/manifest.txt`.
- Справка `references/tooling-installers.md` с общими правилами установщиков
  оснастки.
- Пример адаптера модели `tools/adapters/claude` для Claude Code CLI.

### Изменено

- Канонический источник оснастки модельных прогонов вынесен в
  `scripts/eval-tools/` навыка `ai-apm-setup`; рабочие копии в `tools/` ставит
  установщик.
- В контроль качества `apm run tests` добавлена детерминированная проверка
  расхождения рабочих копий с источником (`check-eval-tools-drift.py`).
- Термины в публикуемой коллекции `.apm/skills/**` унифицированы по `ru-lang`:
  «гейт» → «контроль качества», «эвал» → «модельный прогон», «раннер» →
  «средство запуска», «бандл» → «пакет», «конфиг» → «настройки».

## [0.4.4] - 2026-06-19

### Добавлено

- Команда `apm run evals` для опционального модельного прогона навыков.
- Переносимый контракт адаптера модели и пример адаптера `tools/adapters/codex`.
- Локальный файл настроек моделей `evals.local.yml` вне Git и его образец
  `evals.local.yml.sample`.

### Изменено

- `apm run tests` выполняет только детерминированные проверки, без вызовов
  модели, сети и внешних инструментов, поэтому проверку может выполнить сторонний
  участник и непрерывная интеграция.
- Средство запуска модельных проверок отвязано от Codex CLI и конкретных
  моделей: модель вызывается через адаптер, а модели и судья задаются в формате
  `адаптер:модель` в локальном файле настроек.
- Имена моделей убраны из публикуемой части `.apm/skills/**` и средства запуска.
- Добавлен постоянный сценарий, проверяющий переносимость проверок коллекции.

## [0.4.3] - 2026-06-19

### Изменено

- В `ai-apm-setup` добавлено стоп-условие для проектов, которые только
  потребляют зависимости APM, и добавлены негативные проверки на
  `devDependencies.apm`.
- Уточнены границы между `ai-apm-setup`, `ait-setup` и
  `ai-agents-md-maintenance` для клиентских проектов.

## [0.4.2] - 2026-06-18

### Изменено

- В `apm run tests` добавлена проверка бюджета `description` у навыков.
- Сокращены `description` публикуемых навыков до коротких маршрутизаторов.
- Обновлена зависимость `mekras/ai-dev-team` до `^0.3.1` и перегенерирован
  `apm.lock.yaml`.
- В `.gitignore` добавлен каталог `/.codex`.

## [0.4.1] - 2026-06-18

### Изменено

- Обновлён навык `ai-agents-md-maintenance`: добавлен общий каркас `AGENTS.md`
  и блоки по умолчанию про внесение изменений и Git с сохранением
  пользовательских решений.
- Добавлен сценарий результата, который проверяет, что эти блоки не перетирают
  уже заданную политику проекта.

## [0.4.0] - 2026-06-18

### Изменено

- Исходники публикуемых навыков перенесены в `.apm/skills/**`, а `apm.yml`
  теперь включает `includes: auto` и запускает проверки коллекции из этого
  дерева.

## [0.3.1] - 2026-06-17

### Изменено

- Исключены из Git сгенерированные каталоги `.claude-plugin/` и `build/`.
- Уточнены русские формулировки в навыках без изменения маршрутов и поведения.
- Проверки срабатывания выполняются по навыкам, а сценарии результата — по
  одному, чтобы тяжёлые проверки не превышали тайм-аут из-за объединённого
  контекста.
- Уточнён маршрутизатор `ai-skill-development` для задач про подключение общих
  правил к проектным навыкам.
- Уточнён маршрутизатор `ai-data-collection` для сбора данных перед изменением
  существующих источников истины.
- Уточнён маршрутизатор `ai-apm-setup` для вопросов о необходимости
  `apm compile` в тестовом цикле коллекции навыков.
- Усилен самоконтроль `ai-agents-md-maintenance` после разделения корневых и
  вложенных инструкций.

### Исправлено

- Модельные проверки навыков изолированы от пользовательского `config.toml`.
- Если модель не возвращает результат проверки срабатывания, сценарий
  перезапускается отдельно; неверные классификации по-прежнему считаются
  ошибкой.
- Проверяемая модель теперь получает `expected_output`, `assertions` и
  `must_not`, чтобы сценарии результата проверяли применение навыка, а не
  угадывание скрытых ожиданий.
