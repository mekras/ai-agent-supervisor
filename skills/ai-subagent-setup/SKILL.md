# Маршрутизация подагентов по моделям

Используй этот навык, чтобы спроектировать политику маршрутизации и, при явном
запросе на внедрение, довести её до технической конфигурации конкретной среды.
Цель — стабильное качество при меньшей стоимости через ограниченное
делегирование, проверку и эскалацию.

## Граница навыка

Навык работает в двух режимах:

- `policy` — подготовить черновик политики, риски, контракты и открытые решения
  без изменения файлов.
- `implementation` — создать или обновить реальные файлы конфигурации
  подагентов и связанные проектные инструкции.

Если пользователь просит «спроектируй», «предложи политику», «оцени схему»,
«подготовь черновик» или явно запрещает правки, используй режим `policy`.
Если пользователь просит «настрой», «внедри», «зафиксируй», «создай
конфигурацию», «обнови файлы» или подтверждает черновик, используй режим
`implementation`. Не останавливайся на политике, когда запрос обещает
работающую настройку.

## Обязательный порядок

1. Зафиксируй цель оптимизации: меньшая стоимость, меньшая задержка, большая
   пропускная способность, изоляция контекста, независимая проверка или
   сочетание этих целей.
2. Классифицируй работу по риску, неоднозначности, размеру контекста,
   обратимости, стоимости проверки и ожидаемому типу результата.
3. Реши, оправдано ли делегирование. Не делегируй, если координация и проверка
   с большой вероятностью дороже, чем один проход более сильной модели.
4. Получи список доступных моделей: сначала автоматически, затем простым
   вопросом пользователю только при неудаче автообнаружения.
5. Сам предложи схему использования моделей для обычной работы, узкого поиска,
   ограниченного анализа и редкой сильной проверки. Не проси пользователя
   распределять модели по внутренним ролям.
6. Определи контракт задачи для каждого делегируемого класса: входные данные,
   разрешённые инструменты, область записи, схему результата, свидетельства,
   уверенность, условия остановки и триггеры эскалации.
7. Определи проверки качества перед приёмкой результата: сначала
   детерминированные проверки, затем независимый обзор или сильную модель для
   рискованных изменений.
8. Добавь бюджетные ограничения: максимум токенов, повторов, подагентов,
   контекста и условие возврата к родителю.
9. В режиме `implementation` зафиксируй результат в самом узком долговечном
   месте, которое подходит области действия.
10. Перед финальным ответом проверь, есть ли явный черновик политики или
    ссылка на уже показанный и подтверждённый черновик.

## Правила взаимодействия с пользователем

- Максимально забирай на себя настройку политики, контрактов, проверок и места
  фиксации. Не проси пользователя заполнять матрицы и технические роли, если их
  можно вывести из проекта и запроса.
- Если список моделей неизвестен, сначала попробуй получить его из среды.
  Только при неудаче спроси простой список доступных моделей строками или через
  запятую.
- Если после допустимых немодельных изменений список моделей всё ещё
  неизвестен, финальный ответ должен содержать статус `ожидает список моделей`
  и прямой вопрос о простом списке моделей.
- Для задач настройки Codex один `AGENTS.md` не является полной настройкой:
  нужна реальная конфигурация, закреплённые модели и проверяемый запуск.

## Право на запись подагентов

По умолчанию подагенты работают без записи. Отдельный write-capable подагент на
дешёвой модели допустим только для малых обратимых правок, когда родитель явно
задал файлы или узкую область записи, запретил менять источники истины и может
дёшево проверить diff.

Подагентам запрещено принимать проектные решения, расширять область задачи,
менять правила, навыки, шаблоны, `AGENTS.md`, архитектурные решения, требования
и другие источники истины.

## Формат результата

Для политики или проектного решения включай:

- цель и нецели;
- классы задач и уровни моделей;
- контракты делегирования;
- правила эскалации;
- проверки качества;
- бюджетные ограничения;
- первый эксперимент и план измерения;
- условия пересмотра политики.

Для режима `implementation` дополнительно назови изменённые файлы, слой
фиксации, способ проверки фактического запуска и остаточные открытые решения.

## Ограничения

- Не выдавай политику маршрутизации за техническую настройку.
- Не подставляй конкретные модели без проверки доступности в текущей среде или
  явного списка пользователя.
- Не создавай сложную схему, если доступна только одна или две модели.
- Не добавляй правило отчётности по моделям и лимитам для задач без
  фактического запуска подагентов.
