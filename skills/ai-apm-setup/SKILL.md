# ai-apm-setup

Настраивай проект как коллекцию APM так, чтобы пакет можно было установить,
проверить и выпустить повторяемо. Главный результат — актуальный `apm.yml` и
рабочая команда `apm run tests`, которая запускает тесты коллекции.

## Режим работы

- `setup` — пользователь просит настроить, создать, обновить или исправить
  настройки APM. В этом режиме меняй файлы.
- `audit` — пользователь просит проверить пакет, манифест, выпуск или причину
  сбоя. В этом режиме сначала верни выводы и меняй файлы только по явному
  запросу или если пользователь уже попросил исправить.

## Обязательный порядок

1. Определи корень проекта и проверь, что это коллекция навыков или проект,
   который должен публиковать `skills/**`.
2. Прочитай существующие `apm.yml`, `README.md`, список `skills/**` и доступные
   тестовые команды. Перед правками прочитай
   `references/apm-collection-settings.md`.
3. Если `apm.yml` уже есть, обновляй его с сохранением `name`, `version`,
   `description`, `author`, `license`, `target` и `dependencies`, если
   пользователь не попросил изменить эти поля.
4. Если `apm.yml` отсутствует, создай минимальный манифест коллекции навыков:
   `type: "skill"`, целевые среды из запроса или текущего проекта,
   `dependencies.apm: []`, `dependencies.mcp: []`.
5. Обязательно добавь или обнови `scripts.tests`, чтобы `apm run tests`
   запускал реальные тесты коллекции. Не записывай команду, которая ссылается
   на отсутствующие файлы; сначала найди существующий запуск тестов либо создай
   недостающий проверочный скрипт в рамках задачи.
6. Если подходит стандартная структура этого репозитория, используй команду:

   ```bash
   python3 tools/validate-trigger-evals.py skills --require-all && python3 tools/validate-skill-result-evals.py skills && python3 tools/run-skill-evals.py skills
   ```

7. Для механического создания или обновления базового `apm.yml` можно применить
   `scripts/setup-apm-collection`. После запуска всё равно проверь diff и
   убедись, что команда тестов соответствует реальным файлам проекта.
8. После правки запусти доступные детерминированные проверки: синтаксис
   `apm.yml`, валидаторы тестов навыков, `apm audit` или `apm view`, если они
   доступны локально и не требуют неподходящего окружения.
9. В отчёте назови изменённые файлы, итоговую команду `apm run tests`,
   выполненные проверки и оставшиеся ограничения.

## Ограничения

- Не подменяй настройку APM обычной документацией в README.
- Не перезаписывай существующие метаданные пакета без причины.
- Не добавляй конкретные модели в переносимый навык или манифест коллекции.
- Не считай `apm compile` обязательным для коллекции только из `skills/**`;
  используй его для `instructions` или когда проект явно публикует
  соответствующие примитивы APM.
- Не используй `hooks` и `commands` как основной переносимый механизм, если
  задачу можно решить через `skills`, `prompts`, `instructions` или `agents`.

## Формат результата

Для `setup` укажи:

- что было создано или обновлено в `apm.yml`;
- какая команда стоит в `scripts.tests`;
- чем подтверждено, что `apm run tests` запускает тесты коллекции;
- какие проверки выполнены;
- что осталось проверить человеку перед публикацией.

Для `audit` укажи:

- найденные расхождения по манифесту, тестам и процессу APM;
- риск каждого расхождения;
- достаточное изменение для восстановления работоспособной коллекции.
