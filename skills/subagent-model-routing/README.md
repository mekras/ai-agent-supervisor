# Настройка подагентов и моделей

Навык `subagent-model-routing` используется для настройки экономного
использования подагентов и выбора моделей для разных классов задач.

Есть два уровня результата:

- политика маршрутизации: правило, какие задачи отдавать подагентам, где нужна
  более сильная модель, какие проверки обязательны и когда задача должна
  вернуться основному агенту;
- рабочая настройка: та же политика плюс реальные файлы или настройки среды, в
  которых закреплены подагенты, модели и ограничения запуска.

Одна политика без рабочей настройки не запускает подагентов сама по себе.
Рабочая настройка без политики тоже неудобна: будет непонятно, зачем нужны
подагенты, где их границы и как проверять результат.

Если вы запускаете навык впервые, обычно нужен запрос на рабочую настройку.
Запрос только на политику полезен, когда сначала нужно обсудить подход без
изменения файлов.

## Настройка

**Важно!** Рекомендуется использовать для настройки самую сильную модель, чтобы
получить наилучшие результаты.


Для рабочей настройки можно использовать такой запрос:

```text
Используй $subagent-model-routing.
Настрой экономное использование подагентов для этого проекта.
```

Но лучше указать особенности именно вашего случая. Например:

```text
Используй $subagent-model-routing.
Настрой экономное использование подагентов для этого проекта.
У меня подписка «ChatGPT Pro 5x». Модели gpt-5.4, gpt-5.4-mini и gpt-5.5
используют общие лимиты, но с разной скоростью. У gpt-5.3-codex-spark лимиты
отдельные, её использование не влияет на основные лимиты. 
```

Ожидаемый результат:

- краткая политика маршрутизации;
- рабочая конфигурация подагентов в проекте, если среда её поддерживает;
- явное указание, где именно зафиксирована настройка;
- при необходимости отдельная роль с правом записи для малых обратимых правок,
  а не снятие запрета записи со всех дешёвых подагентов.

Для Codex обычно речь идёт о `.codex/config.toml`,
`.codex/agents/*.toml` и, при необходимости, о правилах использования в
`AGENTS.md`.

Если модели уже выбраны, запрос можно уточнить так:

```text
Используй эти доступные модели: <модель 1>, <модель 2>, <модель 3>.
Сам предложи схему их использования и обнови настройку.
```

Если список моделей не указан, навык сначала пытается получить его сам из
среды. Для Codex он проверяет app-server `model/list`, свежий локальный кэш
моделей и явную конфигурацию. Если список получить не удалось, запрос к человеку
сводится к простому перечню доступных моделей без распределения по ролям.

## Сводка после каждой задачи

Чтобы включить однострочную сводку моделей и лимитов после каждой задачи,
добавьте в `AGENTS.md` одну строку:

```text
После каждой задачи используй `skills/subagent-model-routing` для однострочной сводки моделей и лимитов.
```
## Только политика без правок

Если нужна только политика без файловых правок, лучше использовать явную
формулировку:

```text
Используй $subagent-model-routing.
Подготовь только черновик политики, файлы не меняй.
```

В этом режиме ожидается черновик политики без создания конфигурации
подагентов.

В таком черновике вместо конкретных моделей могут использоваться уровни
`cheap`, `medium`, `strong` как временные обозначения.

В пользовательском ответе назначения объясняются человеческими словами: модель
для обычной работы, быстрая модель для узкого поиска, модель для ограниченного
анализа и сильная модель для редкой проверки.

## Обновление старой настройки

Если подагенты уже настроены старой версией навыка, используйте короткий запрос:

```text
Используй $subagent-model-routing.
Обнови старую настройку подагентов.
```

## Как проверить результат

Для проверки существующей схемы подойдёт такой запрос:

```text
Используй $subagent-model-routing.
Проверь текущую схему подагентов.
```

Для малого эксперимента можно использовать такой запрос:

```text
Проведи эксперимент по использованию подагентов и сравни стоимость, время,
число эскалаций, отклонённые результаты и ошибки после приёмки.
```

Для проверки подагентов Codex нужен технический признак запуска и фактической
модели дочернего потока, а не только текстовый ответ родителя. Подробные
критерии проверки и пример проверочного запроса см. в
`references/routing-patterns.md`.

Настройку стоит считать недостаточной, если выполняется хотя бы одно из условий:

- агент вернул только правила, но не создал поддерживаемую конфигурацию там,
  где она ожидается;
- проверка опирается только на текстовый ответ родителя и не подтверждает
  фактический запуск подагента;
- проверка подтвердила дочерний поток, но не подтвердила фактическую модель
  дочернего запуска;
- нет простого отчёта, по которому видно, какие модели были использованы и
  какой класс лимита они расходовали;
- агент пытается снизить стоимость уменьшением тайм-аута вместо ограничения
  делегирования, моделей, контекста и повторных попыток;
- проверка результата занимает больше времени, чем прямое выполнение сильной
  моделью;
- дешёвая роль с правом записи может менять источники истины или файлы вне
  порученной области;
- подагенты часто эскалируют задачу без полезного промежуточного результата;
- итоговая схема не даёт понять, кто и как принимает результат подагента.

## Внедрение для Codex

Для Codex одного `AGENTS.md` недостаточно: он описывает эксплуатационные правила,
но не создаёт пользовательских агентов и не закрепляет модели. Рабочая настройка
обычно затрагивает `.codex/config.toml`, `.codex/agents/*.toml`, при
необходимости проектные `tools/codex-*` и короткие правила использования в
`AGENTS.md`.

Если запрос ограничен только политикой, `.codex/` файлы могут не появиться, но
тогда результат остаётся черновиком, а не рабочей настройкой.

Технические детали внедрения Codex, примеры TOML, переносимые скрипты, проверка
фактического запуска и отчёты расхода описаны в
`references/routing-patterns.md`.
