# Проверка срабатывания и качества

Используй эту справку, когда меняются триггеры навыка, процедура работы или
признаки успешного результата.

## Проверки срабатывания

Минимальный набор содержит:

- положительные случаи — разные формулировки реальной задачи, где навык нужен;
- отрицательные случаи — близкие задачи, где навык загружаться не должен;
- пограничные случаи — задачи, где нужен соседний навык или локальное правило.

Не путай проверку срабатывания с оценкой качества:

- срабатывание показывает, что нужный навык действительно выбран;
- качество показывает, что выбранный навык даёт проверяемый результат.

В этом проекте проверки хранятся в `evals/triggers.json`:

```json
{
  "skill_name": "example-skill",
  "cases": [
    {
      "id": "example-skill-positive-create",
      "prompt": "Создай навык для проверки договоров.",
      "should_trigger": true,
      "rationale": "Нужно создать навык агента."
    }
  ]
}
```

Структуру проверяй командой:

```bash
python3 tools/validate-trigger-evals.py --require-all
```

Для одного навыка передавай путь к его каталогу:

```bash
python3 tools/validate-trigger-evals.py skills/<name> --require-all
```

## Проверки качества результата

Для реалистичной задачи сформулируй наблюдаемые признаки успеха:

- создан или изменён ожидаемый файл;
- структура соответствует формату навыка;
- `description` покрывает нужные намерения и не захватывает соседние;
- подробности вынесены в подходящие директории;
- выполнены локальные команды проверки;
- отчёт называет остаточный риск.

При анализе качества навыка добавляй отдельный сценарий проверки результата:

- есть ли разница между навыком «до» и «после»;
- какие регрессии появились;
- какой случай проваливается и требует добавления в постоянный набор тестов.

## Контракт проверки применения навыка

Тест результата должен доказывать, что навык не только выбран, но и применён как
рабочая процедура. Для каждого сценария результата фиксируй:

- `evaluation_surface` — какую поверхность проверяет сценарий: маршрутизацию,
  обязательный шаг, ограничение, изменение артефакта, качество результата,
  остановку по правилу или сочетание этих поверхностей;
- `application_evidence` — наблюдаемые признаки применения навыка в ответе,
  артефакте, журнале, diff, трассе выполнения или другом проверяемом следе;
- `oracle` — критерии прохождения и признаки провала, по которым можно
  отличить применение навыка от случайно удачного ответа;
- `negative_control` — что должно провалиться без навыка, без обязательного
  шага, с неверной маршрутизацией или при типичной ошибке. Если такой контроль
  невозможен, явно укажи причину.

Недостаточные проверки не закрывают качество навыка, даже если они полезны как
вспомогательные сигналы:

- проверяется только синтаксис JSON, Markdown, схемы или существование файла;
- проверяется только факт вызова модели, инструмента, скрипта или валидатора;
- проверяется только один внешний признак результата без связи с процедурой
  навыка;
- проверяется только совпадение или отсутствие фрагмента текста;
- проверяется только `should_trigger`, без сценария результата;
- тест мог бы пройти без загрузки навыка или без выполнения его обязательных
  шагов.

Если доступна только такая проверка, называй её вспомогательной и добавляй
постоянный сценарий результата либо фиксируй остаточный риск.

## Быстрый и полный циклы

Скорость проверок оптимизируй выбором минимальной области запуска, но не
сокращением критериев качества.

Используй узкий цикл, когда изменение затрагивает один навык:

```bash
python3 tools/validate-trigger-evals.py skills/<name> --require-all
python3 tools/validate-skill-result-evals.py skills/<name>
python3 tools/run-skill-evals.py skills/<name>
```

Если нужно повторить один модельный сценарий, запускай его по идентификатору:

```bash
python3 tools/run-skill-evals.py skills/<name> --case-id <id>
```

Через APM тот же выборочный запуск задаётся переменными среды:

```bash
APM_EVAL_PATH=skills/<name> apm run tests
APM_EVAL_PATH=skills/<name> APM_EVAL_CASE_ID=<id> apm run tests
```

Расширяй запуск до всей коллекции, если:

- изменён `description` и есть риск влияния на соседние навыки;
- изменены общие валидаторы, средство запуска или `apm.yml`;
- изменены несколько навыков;
- результат готовится к публикации или выпуску пакета;
- узкий запуск выявил сбой, причина которого может быть общей.

Выборочный запуск должен быть отражён в отчёте: укажи путь или `id`, что именно
он покрывает, какие проверки не запускались и почему этого достаточно для
текущего шага. Перед публикацией не заменяй полный прогон одним выбранным
сценарием.

## Виды тестов навыка

При создании, изменении или сопровождении навыка сначала проверь, есть ли у
него постоянные тесты и покрывают ли они текущий риск. Если тестов нет, их не
хватает или текущая правка меняет поведение навыка, добавь или обнови один или
несколько видов тестов. Если постоянный тест сейчас нельзя создать, в отчёте
явно укажи, какой тест нужен, почему он не добавлен и какой риск остаётся.

Не заменяй создание доступного теста предложением добавить его позже: предложение
допустимо только при недоступном контексте, отсутствии проверочной
инфраструктуры или явном ограничении области задачи.

- Тесты срабатывания: положительные, отрицательные и пограничные запросы в
  `evals/triggers.json`.
- Тесты результата: сценарий с входным запросом, ожидаемыми артефактами,
  признаками применения навыка, oracle, негативным контролем и остаточными
  рисками.
- Регрессионные тесты: случай из реального сбоя или спорного решения, который
  должен продолжать проходить после изменения навыка.
- Негативные тесты ограничений: сценарии, где навык обязан остановиться,
  передать решение человеку или выбрать соседний навык.
- Тесты материалов навыка: проверка существования ссылок на `references/`,
  `assets/`, `scripts/`, `templates/` и соответствия этих материалов
  навигации из `SKILL.md`.
- Тесты скриптов: запуск или сухая проверка команд из `scripts/`, если навык
  полагается на повторяемую механику.
- Интеграционные тесты проекта: сценарий совместного применения навыка с
  `AGENTS.md`, соседними навыками и локальными правилами целевого проекта.

Минимум для нового навыка: тесты срабатывания и один тест результата. Минимум
для изменения `description`: обновлённые положительные, отрицательные и
пограничные случаи. Минимум для изменения процедуры или ограничений: один тест
результата и один негативный тест ограничения.

## Аудит покрытия перед завершением

Перед завершением изменения навыка проверь:

- какие тесты уже были у сопровождаемого навыка;
- какие файлы навыка изменены: `SKILL.md`, `agents/openai.yaml`, `references/`,
  `assets/`, `scripts/`, `evals/`;
- какой риск изменения появился или изменился;
- какие постоянные тесты добавлены или обновлены под этот риск;
- какие проверки запущены и что они подтвердили.

Если изменение затрагивает только текст справки, всё равно проверь, не меняет ли
оно процедуру, ограничения, критерии результата или маршрутизацию. Если меняет,
постоянные тесты нужны так же, как при изменении `SKILL.md`.

## Когда нужны дополнительные проверки

- изменено `description`;
- изменён `SKILL.md`, `agents/openai.yaml`, процедура, ограничения, критерии
  результата или справка, влияющая на поведение агента;
- добавлен новый соседний навык;
- реальная задача показала неверное срабатывание;
- навык стал длиннее или получил новые справки;
- добавлен скрипт, который может зависеть от окружения;
- проектный `AGENTS.md` изменил правила работы с навыками.
- пользователь просит создать, проверить или сопровождать тесты навыка.

## Ловушки

- Один удачный пример не доказывает качество маршрутизации.
- Проверка, которая проходит без навыка, не доказывает пользу навыка.
- Модельный запуск сам по себе не доказывает применение навыка.
- Косвенный сигнал ответа не заменяет проверку поверхности применения навыка.
- Нельзя подгонять `description` только под текущий набор примеров.
- Модель-судья полезна только вместе с фактами, файлами, выводами команд или
  другим проверяемым артефактом.
