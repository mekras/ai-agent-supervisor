# project-rules-audit

Проверяй правила проекта как рабочую систему управления агентом: они должны
помогать агенту действовать точнее, не конфликтовать между собой и не тратить
контекст без пользы.

## Назначение

Навык нужен для глубокого аудита набора правил, которые управляют поведением
агента в репозитории. Он шире, чем проверка одного `AGENTS.md`: учитывает
несколько форматов инструкций, локальные и глобальные навыки, символические
ссылки, README с указаниями для ИИ и проектные правила, которые могут
конфликтовать между собой.

## Основания корпуса знаний

Навык опирается на локальный корпус:

- `knowledge/ai-control-layer.md` — проектный слой управления ИИ и контроль
  применения правил;
- `knowledge/statements/CODX.md` — обнаружение и объединение `AGENTS.md`,
  область действия и риск лимита контекста;
- `knowledge/statements/GHAB.md` — ценность конкретных инструкций, команд,
  границ, ролей и примеров;
- `knowledge/statements/GIST.md` — разделение машинных инструкций и README,
  риски секретов, раздувания и устаревания;
- `knowledge/statements/PHSC.md` — вред перегруженных инструкций и польза
  структуры `WHAT/WHY/HOW`;
- `knowledge/statements/VSCD.md` — несколько файлов инструкций и риск
  неопределённого порядка применения;
- `knowledge/statements/ASKL.md` и `knowledge/statements/PPRX.md` — правила
  разработки навыков, проверки триггеров и сопровождения по реальным ошибкам.

Положения корпуса являются основанием для проверки, но не заменяют чтение
текущих файлов проекта.

## Когда применять

Применяй навык, если нужно:

- проверить правила проекта для агента глубже, чем обычный обзор одного файла;
- найти дубли, противоречия, неоднозначности, устаревшие места и лишнюю
  стоимость контекста;
- сравнить несколько источников инструкций для ИИ;
- упорядочить правила между `AGENTS.md`, другими файлами инструкций, навыками и
  README;
- подготовить минимальные правки правил после найденных проблем;
- сопровождать набор правил в разных репозиториях.

Если задача ограничена только качеством одного или нескольких `AGENTS.md` без
широкого аудита правил, применяй `skills/agents-md-review/SKILL.md`. Если нужно
создать, изменить, сократить или разделить `AGENTS.md`, применяй
`skills/agents-md-maintenance/SKILL.md`. Если задача меняет сам навык, применяй
`skills/skill-development/SKILL.md`.

## Обязательный порядок

1. Уточни цель проверки, только если без этого нельзя безопасно продолжить:
   найти проблемы, предложить правки, внести изменения, сравнить текущее
   состояние с предыдущим или выдать полный отчёт.
2. Найди источники правил. Начни с `AGENTS.md` и известных файлов инструкций
   для агентов, затем ищи локальные правила по словам `agent`, `AI`,
   `assistant`, `ИИ`, `instructions`, `rules`, `prompt`, `policy`,
   `guidelines`. Если проект использует навыки, учитывай локальные, глобальные
   и подключённые символическими ссылками навыки, доступные агенту или явно
   упомянутые в проекте.
3. Построй карту правил: файл, область действия, адресат, приоритет,
   повторяемые темы, обязательность, срок жизни и связь с инструментами.
4. Проверь проблемы: дубли, противоречия, неоднозначности, непроверяемые
   требования, устаревшие сведения, слишком широкие запреты, разрастание,
   смешение проектных и переносимых правил, локальные копии глобальных правил
   или правил из символических ссылок, лишнюю нагрузку на контекст.
5. Самостоятельно перепроверь выводы: для каждого существенного замечания
   найди подтверждение в текущем содержимом файлов или явно пометь его как
   предположение. Не используй в доказательствах строки из памяти, старых
   выводов или уже изменённых версий файлов.
6. По умолчанию разбирай найденные проблемы по одной: сначала собери
   достаточно контекста для ранжирования, затем покажи только одну наиболее
   важную проблему, варианты решения и рекомендацию. После этого остановись и
   жди решения пользователя; после одобрения, отклонения или уточнения по
   текущей проблеме сразу покажи следующую проблему с вариантами решения и
   рекомендацией. Не спрашивай отдельно, продолжать ли разбор, пока не
   закончились найденные проблемы или пользователь не попросил остановиться.
   После отработки последней найденной проблемы явно сообщи, что все найденные
   проблемы отработаны. Полный список проблем выдавай только по явной просьбе
   пользователя.
7. Если пользователь сразу просит исправить правила, вноси минимальные правки,
   сохраняя смысл, приоритеты и локальные соглашения проекта.
8. После правок проверь результат: перечитай изменённые места, убедись, что
   новые формулировки не создают новых конфликтов, и назови остаточные риски.

## Что читать дополнительно

- `references/audit-checklist.md` — полный проверочный список и формат отчёта.

## Границы

- Не превращай аудит в переписывание всех правил с нуля, если цель —
  диагностика.
- Не удаляй правило только потому, что оно повторяется: сначала проверь, не
  задаёт ли повтор приоритет, область действия или важное усиление.
- Удаляй локальный дубль глобального или подключённого через символическую
  ссылку навыка только после проверки, что внешний источник реально доступен и
  локальная копия не добавляет проектной области действия.
- Не объединяй проектные правила с переносимыми без явного основания.
- Не цитируй и не пересказывай правило как доказательство, пока не проверил,
  что оно дословно или по смыслу присутствует в текущем снимке указанного
  файла. Если строка исчезла после правок, перечитай файл и обнови вывод.
- Не считай найденный конфликт доказанным, пока не указаны обе конфликтующие
  формулировки или их точные места.
- Не добавляй новые правила «для полноты», если они не закрывают найденный риск
  или реальную ошибку агента.

## Вывод

По умолчанию не выдавай полный список проблем. Покажи одну наиболее важную
проблему, предложи варианты решения и дождись явного решения пользователя.
После решения по текущей проблеме сразу переходи к следующей. Если последняя
найденная проблема отработана, явно сообщи, что все найденные проблемы
отработаны.

Если пользователь явно просит полный отчёт, сначала перечисляй проблемы и
риски, затем предлагаемые правки, после этого краткое резюме. Для каждой
проблемы указывай файл и строку, если они доступны, тип проблемы, влияние на
работу агента и конкретное действие.
