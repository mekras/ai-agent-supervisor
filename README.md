# Коллекция навыков для управления применением ИИ

Коллекция переносимых навыков для контроля использования ИИ в проекте.
Обеспечивает системное, безопасное и проверяемое использование ИИ в разработке
ПО. Задача этих навыков — защищать человека и проект от неправильного или
неподходящего применения ИИ.

Коллекция ориентирована прежде всего на `Codex`, `Claude Code` и совместимые
среды, которые умеют подключать локальные навыки через `APM`.

Коллекция задаёт переносимый слой управления применением ИИ в проектах. Этот
слой помогает:

- заранее отделять задачи, которые можно поручать ИИ, от задач с недостаточным
  контекстом или слишком высоким риском;
- контролировать изменения правил, ролей, навыков, шаблонов, документации и
  других источников истины;
- собирать проверяемые основания до проектирования переносимых решений;
- принимать или отклонять результаты ИИ перед публикацией и дальнейшим
  использованием;
- разбирать нарушения правил как возможный дефект правил, маршрутизации или
  контрольной процедуры.

## Установка и подключение

Основной способ использования — установка коллекции через
[APM](https://github.com/microsoft/apm). После установки навыки коллекции
становятся доступны агенту как обычные локальные навыки.

Чтобы установить коллекцию выполните в корне вашего проекта:

```bash
apm install --dev github.com/mekras/ai-agent-supervisor#^0.5.2
```

Обновить уже установленную коллекцию:

```bash
apm update
```

## Использование

Навыки коллекции разрабатываются так, чтобы в идеале пользователю вообще не
нужно было про них помнить и как-то явно использовать. Цель в том, чтобы навыки
самостоятельно включались в нужных случаях и выполняли предназначение коллекции.
Однако возможно и ручное их использование, например, для уверенности, что навык
был применён или в ососбых случаях, когда агент не понимает, что нужно применить
навык из этой коллекции.

### Проверка правил проекта

Для всесторонней проверки агентских правил проекта используйте навык
[ai-audit-project](.apm/skills/ai-audit-project/README.md).

## Перечень навыков

- `ai-agents-md-maintenance` — создание и сопровождение `AGENTS.md`.
- `ai-setup-apm` — настройка проекта как коллекции навыков APM.
- `ai-application-check` — проверка применимости ИИ к задаче.
- `ai-audit-agents-md` — проверка качества `AGENTS.md`.
- [ai-audit-project](.apm/skills/ai-audit-project/README.md) — аудит правил проекта
  для агента.
- `ai-data-collection` — порядок сбора данных перед проектированием изменений.
- `ai-rule-failure-analysis` — разбор нарушения правил агентом как дефекта
  правил или контроля.
- `ai-skill-development` — создание, проверка и сопровождение навыков агента.
- `ai-work-control` — контрольный регламент перед существенным применением ИИ.
- `ai-work-result-evaluation` — проверка результата перед публикацией и
  приёмкой.
- [ai-setup-subagents](.apm/skills/ai-setup-subagents/README.md) — настройка экономной
  маршрутизации подагентов и моделей с проверками качества.

## Разработка

Для проверки части, связанной с выпуском пакета, перед публикацией используйте
короткий цикл:

```bash
apm install /path/to/ai-agent-supervisor
apm view ai-agent-supervisor
apm audit
apm pack
```

`apm view` и `apm audit` удобнее запускать в отдельном тестовом проекте, куда
пакет установлен локально. Это позволяет увидеть пакет так, как его увидит
потребитель.

Если в будущем пакет начнёт публиковать `instructions`, перед выпуском
добавляйте ещё два шага:

```bash
apm compile --validate
apm compile --dry-run --target claude,codex
```

### Проверки

Проверки разделены на две команды.

Обязательная проверка не вызывает модель, сеть или внешние инструменты, поэтому
её может выполнить любой участник и непрерывная интеграция:

```bash
apm run tests
```

Она проверяет бюджет `description` и структуру проверок срабатывания и
результата.

Модельный прогон — отдельная необязательная команда. Это измерение качества, а не
условие приёмки:

```bash
apm run evals
```

Он применяет навык к сценариям: модель выбирает срабатывание по
`evals/triggers.json`, применяет навык к сценариям из
`evals/result-scenarios.json`, а модель-судья сверяет ответы с ожиданиями
сценария.

#### Настройка модельного прогона

Средство запуска вызывает модель через адаптер по переносимому контракту: вызов
`<команда адаптера> <модель>`, запрос на стандартном вводе, текст ответа на
стандартном выводе. Готовый пример для Codex CLI — `tools/adapters/codex`. Под
другую модель напишите свой адаптер по тому же контракту.

Адаптеры, список моделей и модель-судью задаёт локальный файл настроек
`evals.local.yml`. Он не попадает в Git: средство запуска создаёт его из
`evals.local.yml.sample` при первом запуске `apm run evals` и добавляет в
`.git/info/exclude`. В разделе `adapters` задайте соответствие имени адаптера и
команды, а модели и судью указывайте в формате `адаптер:модель`. Ключи и подписку
передавайте через переменные окружения адаптера, а не через этот файл.

```yaml
adapters:
  codex: tools/adapters/codex
  # claude: tools/adapters/claude
models:
  - codex:gpt-5.3-codex-spark
  # - claude:opus
judge: codex:gpt-5.5
timeout: 900
```

Формат `адаптер:модель` позволяет в одном прогоне сочетать модели от разных
адаптеров. Каждая запись проверяется по одинаковым критериям, поэтому результаты
сопоставимы.

#### Выборочный запуск

Для быстрой проверки одного навыка:

```bash
APM_EVAL_PATH=.apm/skills/ai-skill-development apm run evals
```

Для повторного запуска одного модельного сценария:

```bash
APM_EVAL_PATH=.apm/skills/ai-skill-development APM_EVAL_CASE_ID=ai-skill-development-result-stale-existing-tests apm run evals
```

Для ограниченной локальной проверки можно временно сократить число сценариев
результата:

```bash
APM_EVAL_LIMIT=1 apm run evals
```

Перед публикацией пакета запускайте полный `apm run evals` без этих ограничений.
