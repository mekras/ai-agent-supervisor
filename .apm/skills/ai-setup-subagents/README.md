# Настройка подагентов и моделей

`ai-setup-subagents` помогает настроить экономное использование подагентов:
какие задачи отдавать быстрой модели, где нужна проверка, а где лучше оставить
работу основной модели.

Навык полезен, если вы хотите:

- меньше тратить основной лимит модели;
- не настраивать одних и тех же подагентов в каждом проекте заново;
- сохранить единые правила проверки результата;
- оставить возможность особых правил в отдельных проектах.

## Как попросить

Для обычной настройки:

```text
Используй $ai-setup-subagents.
Настрой экономное использование подагентов.
```

Если одна схема нужна в нескольких проектах на одной машине:

```text
Используй $ai-setup-subagents.
Настрой общую схему подагентов для моих проектов.
```

Если вы уже знаете доступные модели, можно перечислить их сразу:

```text
Используй $ai-setup-subagents.
Настрой общую схему подагентов для моих проектов.
У меня подписка «ChatGPT Pro 5x». Модели обычная-модель, экономичная-mini-модель и сильная-модель
используют общие лимиты, но с разной скоростью. У модель-с-отдельным-лимитом лимиты
отдельные, её использование не влияет на основные лимиты. 
```

Если модели не указаны, агент сначала попробует найти их сам. Если не
получится, попросит список моделей.

## Где хранить настройку

Для повторяющейся схемы лучше использовать пользовательскую настройку активной
оснастки, а в проектах хранить только локальные отличия.

Так меньше копирования, меньше расхода токенов на повторную настройку и меньше
риск, что разные проекты постепенно разъедутся по правилам.

Если у оснастки нет подходящего пользовательского слоя, настройка создаёт
`subagents.local.toml` и исключает его через `.git/info/exclude`. В Git такая
политика попадает только после отдельного согласия пользователя с названием
файла. В проекте остаются только локальные переопределения: особые модели,
запреты, команды проверки или правила приёмки.

Название и версию схемы агент подберёт сам и покажет в результате настройки.

## Как обновить

Если подагенты уже настроены, попросите обновить схему через текущую версию
навыка:

```text
Используй $ai-setup-subagents.
Обнови существующую настройку подагентов.
```

Агент сам проверит текущую схему, найдёт отслеживаемые настройки прежнего
формата и предложит перенести их в локальный слой. Он не изменит файл в Git без
отдельного согласия.

## Настройка одного проекта

Проектная настройка нужна, когда правила действительно отличаются от остальных
проектов: например, подагентам нельзя писать файлы, нужны другие модели или
есть особые источники истины.

Для Codex основной слой рабочей настройки — пользовательская конфигурация
`~/.codex`; для другой оснастки используй её пользовательский слой. Если такого
слоя нет, применяй неотслеживаемый `subagents.local.toml`. Короткие правила в
`AGENTS.md` описывают только порядок чтения настройки в конкретном проекте.

## Как проверить

Для проверки текущей схемы:

```text
Используй $ai-setup-subagents.
Проверь текущую схему подагентов.
```

Настройка считается неполной, если есть только текстовые правила, нет рабочего
локального слоя либо хотя бы одна назначенная модель не прошла короткую проверку
через адаптер своей оснастки. Проверка модели и проверка реального запуска
подагента различаются: для запуска дополнительно нужен журнал и отчёт расхода.

Технические детали, шаблоны и порядок проверки разных оснасток описаны в
`references/routing-patterns.md`.

Сначала получите каталог активной оснастки, затем назначьте роли точным
идентификаторам из него. Название семейства не запускает модель. Для проекта с
историей полезен обезличенный анализ JSONL: он считает только агрегаты и не
выводит тексты запросов. Проверка доступности модели не заменяет журнал запуска
роли: наследование модели родителя, отсутствие роли или модели означает провал.
