# Шаблоны маршрутизации

Используй эти шаблоны как отправную точку. Адаптируй их к проекту, а не
копируй без изменений.

## Матрица классификации задач

| Измерение | Делегирование дешёвой модели подходит, когда | Нужна эскалация, когда |
|---|---|---|
| Риск | Результат рекомендательный, обратимый или будет проверен позже | Результат меняет долговечное знание проекта |
| Неоднозначность | Критерии явные и локальные | Цель, область или критерии приёмки неясны |
| Контекст | Входные данные малы и самодостаточны | Правильность зависит от широкой истории репозитория |
| Проверка | Схема, тесты, ссылки или различия позволяют проверить результат | Нужно суждение человека или сильной модели |
| Цена ошибки | Плохой результат легко отбросить | Плохой результат может ввести в заблуждение будущие решения |
| Независимость | Части не влияют друг на друга | Подзадачам нужно общее изменяющееся состояние |

## Пример таблицы маршрутизации

| Класс задачи | Рекомендуемый уровень | Ответственность родительского агента |
|---|---|---|
| Найти кандидаты: файлы, заметки или источники | `cheap` | Проверить риск неполноты и выбрать, что читать |
| Извлечь утверждения из одного источника | `cheap` | Проверить выборку и отклонить неподтверждённые утверждения |
| Суммировать много независимых фрагментов | `cheap` | Объединить, найти противоречия, сохранить ссылки |
| Выполнить малую обратимую правку в указанных файлах | `cheap` с правом записи | Проверить различия, запустить проверку и принять результат |
| Сравнить два ограниченных документа | `medium` | Решить влияние и нужные изменения |
| Подготовить черновик требований из проверенных фактов | `medium` | Проверить полноту и критерии приёмки |
| Изменить документацию как источник истины | `strong` | Отвечать за итоговую формулировку и согласованность |
| Изменить правила проекта или навыки | `strong` | Отвечать за метод, проверку и приёмку |
| Принять архитектурное или политическое решение | `strong` | Отвечать за компромиссы и условия пересмотра |

## Контракт делегированной задачи

Используй компактный контракт:

```text
Цель:
Граница входных данных:
Разрешённые действия/инструменты:
Разрешённая область записи:
Запрещённые действия:
Схема результата:
Обязательные свидетельства:
Шкала уверенности:
Остановиться и эскалировать, если:
Бюджет токенов/времени:
```

Предпочитай маленькие явные контракты широким поручениям. Если контракту нужно
длинное объяснение, задачу может быть невыгодно делегировать дешёвой модели.

## Пример схемы результата

```json
{
  "status": "complete|partial|blocked",
  "items": [
    {
      "claim": "Короткое фактическое утверждение или кандидат результата",
      "evidence": "Путь к файлу, строка, URL или идентификатор источника",
      "confidence": "high|medium|low",
      "risk": "Что может быть неверно"
    }
  ],
  "open_questions": [],
  "escalation_needed": false
}
```

## Триггеры эскалации

- Не хватает обязательных входных данных.
- Источники противоречат друг другу.
- Результат не может сослаться на источники.
- Потребуется больше одной повторной попытки.
- Подагенту нужно менять файлы вне своей области.
- Задача требует суждения о правилах, архитектуре, безопасности, праве,
  финансах или источнике истины.
- Подагенту с правом записи нужно выбирать новые файлы для записи или менять
  источники истины.
- Результат выглядит правдоподобно, но его нельзя дёшево проверить.

## Бюджетные ограничения

- Задавай максимум подагентов на одну родительскую задачу.
- Задавай максимум размера контекста для делегированной задачи.
- Предпочитай один дешёвый проход с проверкой повторным дешёвым попыткам.
- Считай полную стоимость, а не только цену токенов модели.
- Включай стоимость проверки в решение о маршрутизации.
- Отключай делегирование для микрозадач, где накладные расходы поручения
  преобладают.
- Не придумывай универсальные числовые пороги. Используй проектные измерения,
  если они есть; иначе начинай с консервативных ограничений и пересматривай их
  после малого эксперимента.

## Сопоставление уровней с моделями

Для черновика политики используй названия уровней, пока конкретные модели не
известны. Для внедрения сначала попробуй получить список моделей из текущей
среды Codex или другого целевого окружения. Не бери модели из документации,
глобальных настроек или другого проекта как готовое решение. Если
автообнаружение не сработало, спроси у пользователя простой список доступных
моделей и сам сопоставь их с назначениями. Если модели не известны, фиксируй
открытое решение вместо молчаливой подстановки.

```text
cheap:
  model:
  strengths:
  forbidden_for:
  max_context:
  max_retries:

medium:
  model:
  strengths:
  forbidden_for:
  max_context:
  max_retries:

strong:
  model:
  responsibilities:
  must_review:
```

Если в проекте несколько дешёвых или средних моделей, разделяй их по
возможностям, а не только по цене: извлечение, код, чтение длинного контекста,
многоязычная работа, использование инструментов или надёжность
структурированного результата.

## Пример Codex-конфигурации

Для Codex политика маршрутизации не заменяет техническую настройку агентов
Codex. В режиме внедрения проверь или создай проектные файлы `.codex/`.
Если `.codex/` отсутствует, создай настройку с нуля, а не останавливайся на
инструкции пользователю.

Если одинаковая схема подагентов нужна в нескольких проектах на одной машине,
сначала проверь глобальный слой Codex: `$CODEX_HOME`, `~/.codex/config.toml`,
профили `~/.codex/<profile>.config.toml` и поддерживаемое место для глобальных
определений агентов в текущей версии Codex. Общие параметры и стандартные роли
лучше держать глобально, а в проекте оставлять только переопределения.

Если глобального слоя ещё нет, но в текущем проекте уже есть рабочая `.codex/`
схема, предложи перенос:

1. Выделить общие параметры fan-out, стандартные роли, модели и правила
   проверки.
2. Перенести их в глобальную конфигурацию или глобальный профиль Codex.
3. Оставить в проекте только локальные отличия и короткую ссылку на глобальную
   схему.
4. Проверить фактический запуск подагента после переноса по JSONL и строке
   расхода.

Сначала покажи этот перенос человеку как согласуемый черновик прямо в ответе.
Не создавай отдельный документ проекта для плана переноса, если человек явно не
попросил сохранить план в файле. После согласования применяй изменения к
конфигурации и проверяй реальный запуск.

Для версии схемы сам выбери короткое имя по области действия и источнику
настройки. Не проси человека придумывать служебный идентификатор. Покажи
выбранное имя и версию в человекочитаемой записи:

```text
Схема подагентов: <имя>, версия <дата или версия>.
Источник: <источник>.
```

Минимальные ограничения fan-out:

```toml
# .codex/config.toml
[features]
# Включай только если текущий Codex CLI показывает, что реальный fan-out
# находится за этим флагом.
enable_fanout = true

[agents]
max_threads = 2
max_depth = 1
# Не сокращай ради скорости, если цель — стоимость.
job_max_runtime_seconds = 1800
```

`max_threads` ограничивает число одновременных потоков. `max_depth = 1`
запрещает рекурсивный fan-out: дочерний агент не должен порождать следующий
уровень подагентов без отдельного решения.

Пример дешёвого read-only исследователя:

```toml
# .codex/agents/cheap_explorer.toml
name = "cheap_explorer"
description = "Read-only explorer for low-risk evidence gathering."
model = "<cheap-model>"
model_reasoning_effort = "low"
sandbox_mode = "read-only"
developer_instructions = """
Work only on narrow read-only tasks.
Search and read the minimum required files.
Return concise structured findings with file references.
Do not edit files.
Do not decide policy.
Escalate on ambiguity, conflicting evidence, missing context, or requests to
change project sources of truth.
"""
```

Пример дешёвого исполнителя для малых обратимых правок:

```toml
# .codex/agents/cheap_patch_worker.toml
name = "cheap_patch_worker"
description = "Cheap bounded patch worker for small reversible edits requested by the parent agent."
model = "<cheap-code-model>"
model_reasoning_effort = "low"
sandbox_mode = "workspace-write"
developer_instructions = """
Edit files only when the parent task gives an explicit file list or a narrow
write scope.
Make only small reversible changes with a cheap verification path.
Return changed files, summary, verification performed or recommended, risk, and
escalation_needed.
Do not edit AGENTS.md, skills, rules, templates, requirements, architecture
decisions, or other sources of truth.
Do not broaden the task, choose unrelated files, accept the final result, or make
project decisions.
Escalate on ambiguity, missing write scope, failing verification, broad context,
or requests to change project sources of truth.
"""
```

Не превращай `cheap_explorer` в роль с правом записи. Старые проекты обновляй
добавлением отдельного patch-worker и сужением общего запрета записи до правила
по умолчанию.

Пример среднего проверяющего для ограниченных сравнений:

```toml
# .codex/agents/bounded_policy_reviewer.toml
name = "bounded_policy_reviewer"
description = "Reviewer for bounded policy drafts and routing consistency."
model = "<medium-model>"
model_reasoning_effort = "medium"
sandbox_mode = "read-only"
developer_instructions = """
Review only the provided policy draft and referenced files.
Check consistency, missing escalation rules, and unsupported claims.
Return risks, required evidence, and suggested corrections.
Do not accept the final result.
Do not edit files.
Escalate if the review requires a project decision or broad context.
"""
```

Для Codex один `AGENTS.md` без пользовательской конфигурации `~/.codex` или
обоснованного проектного слоя `.codex/` не является полной настройкой агентов
Codex. Конкретная политика подагентов должна фиксироваться в целевом проекте
или пользовательском слое, а не в переносимом навыке.

### Фиксация политики в целевом проекте

Используй этот раздел как проверку результата настройки, а не как готовую
политику. Для каждого целевого проекта отдельно реши:

- где живёт рабочая конфигурация подагентов;
- где зафиксированы проектные классы задач для делегирования;
- какие роли, модели, лимиты и запреты доступны именно в этой среде;
- какой технический след доказывает запуск и переключение модели;
- какой отчёт нужен после фактического запуска подагентов.

Если политика фиксируется в `AGENTS.md`, применяй профильный навык сопровождения
`AGENTS.md`. `ai-setup-subagents` отвечает за содержание политики подагентов, но
не заменяет общие правила минимальности, иерархии и машинной пригодности
`AGENTS.md`.

### Ясность обязательности делегирования

В проектных правилах разделяй три уровня обязательности:

| Уровень | Как формулировать | Последствие |
|---|---|---|
| Не применять | Микрозадача или проверка дешевле прямым проходом родителя | Родитель выполняет сам |
| Предпочтительно | Узкий поиск, извлечение фактов или проверка охвата с дешёвой проверкой результата | Родитель пытается делегировать, но может обоснованно не делегировать |
| Обязательно | Пользователь или проектное правило прямо требуют подагента, либо задача нужна для проверки самой схемы подагентов | Недоступность роли, модели, лимита или технического следа блокирует задачу до разрешения человека |

Не смешивай «предпочтительно» и «обязательно» в одном абзаце. Если недоступность
подагента должна блокировать задачу, укажи, какое условие делает запуск
обязательным и какой технический след нужно проверить.

## Обновление старой настройки

Для проекта, настроенного старой версией навыка, используй простой порядок:

1. Найди пользовательский слой `~/.codex/config.toml`, доступные глобальные
   определения агентов, при наличии проектные `.codex/config.toml`,
   `.codex/agents/*.toml`, `AGENTS.md` и `tools/codex-*`.
2. Оставь роли только для чтения (`cheap_explorer`, роли проверки) без права записи.
3. Добавь отдельную роль `cheap_patch_worker` или аналог с `workspace-write`.
4. Замени общий запрет «подагенты не редактируют файлы» на правило по умолчанию
   с исключением для роли с правом записи по явному поручению родителя.
5. Обнови запускатель: по умолчанию `read-only`, для patch-worker только явный
   `CODEX_SUBAGENT_SANDBOX=workspace-write`.
6. Проверь, что источники истины остаются запрещены для дешёвых подагентов
   с правом записи, а родитель принимает результат по различиям и проверке.

Человеку достаточно попросить: «Обнови старую настройку
`ai-setup-subagents` до версии с отдельным дешёвым patch-worker». Остальное
агент должен вывести из файлов проекта.

## Маршрут через отдельный процесс для выбора модели

Если проверка показывает, что встроенный fan-out создаёт дочерний поток, но не
применяет модель из пользовательского или проектного слоя Codex, используй
отдельный процесс Codex для задач, где модель должна быть выбрана
гарантированно.

Переносимые запускатели лежат в директории `scripts/` навыка:

- `codex-model-subagent` — запуск выбранной модели;
- `codex-usage-report` — краткий, однострочный, подробный и JSON-отчёт;
- `install-codex-tools` — копирование скриптов в `<project-root>/tools`.

Установщик следует общим правилам установщиков оснастки (навык `ai-setup-apm`,
`references/tooling-installers.md`): вызов `install-* [project-root]`,
копирование в `tools/`, строка `installed=` в конце.

При внедрении в проект выполни установку из директории навыка:

```bash
.apm/skills/ai-setup-subagents/scripts/install-codex-tools .
```

После установки вызов выглядит так:

```bash
tools/codex-model-subagent экономичная-mini-модель bounded-check \
  'Return a short JSON result for the bounded task.'
```

Если среда APM даёт стабильный путь к файлам навыка, можно вызывать
`scripts/codex-model-subagent` напрямую без копирования в проект.

После задачи, где фактически запускались подагенты, однострочный отчёт должен
быть понятен без чтения JSONL:

```text
Модели: экономичная-mini-модель 14.5 тыс. | лимиты: общий 14.5 тыс. | время: 6.9 с
```

Для подробного анализа сохраняй JSONL и финальный ответ. Отдельно проверяй,
какие модели относятся к общему лимиту, какие имеют отдельный лимит, и какие
модели расходуют общий лимит медленнее.

## Проверочный запрос для Codex

Короткая просьба к родителю может не запустить подагента, потому что родитель
вправе решить, что координация дороже прямого выполнения. Для проверки
используй явный контракт:

```text
Follow the current project subagent scheme.
This task should use `cheap_explorer` for bounded evidence gathering if
justified, and may use `strong_reviewer` for an independent second pass if that
is justified by the current policy. Parent agent must make the final acceptance
decision.

Task: review the current subagent scheme using only `.codex/config.toml`,
`~/.codex/config.toml`, user or global agent definitions, optional
project `.codex/agents/*.toml`, `AGENTS.md`, and `ai-setup-subagents`.

Return valid JSON only with keys:
`findings`, `used_subagents`, `escalations`, `rejected_results`.
```

Проверка успешна только при техническом признаке запуска подагента:
`thread_spawn_edges`, `subagent_notification`, дочерний поток в состоянии Codex
или JSONL-событие subagent. Ответ родителя без такого признака не доказывает
работу подагентов.

Не используй `codex exec --ephemeral` как обязательную проверку Codex
subagents. В этом режиме app-server может быть недоступен, из-за чего
порождение дочерних потоков проверяется неправильно.

`/agent` показывает активные потоки агентов. Он может показывать только
`Main [default]`, если дочерние потоки не запущены; это не список файлов
`.codex/agents/*.toml`.

## План измерения

Прогони политику на малом наборе реальных задач и зафиксируй:

- общее число токенов и вызовов инструментов;
- время выполнения;
- число эскалаций;
- число отклонённых результатов подагентов;
- дефекты, найденные после приёмки;
- время проверки родительским агентом;
- была бы та же задача дешевле при прямом проходе сильной моделью.

Пересматривай таблицу маршрутизации, когда дешёвые результаты часто требуют
исправления, эскалации становятся частыми или проверка занимает больше времени,
чем прямое выполнение.
