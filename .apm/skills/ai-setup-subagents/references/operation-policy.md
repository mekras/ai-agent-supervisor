# Рабочая политика настройки подагентов

Используй эту справку для деталей, которые не должны раздувать `SKILL.md`.

## Работа с пользователем

Сначала изучи доступный проектный контекст: README, `AGENTS.md`, существующие
навыки, правила, шаблоны, команды, документацию процесса и явные сведения о
моделях. Не проси пользователя заполнять матрицы и контракты вручную, если их
можно вывести из проекта и запроса.

Модели — обязательная входная информация, но не ручная классификация для
человека. Сначала сам попробуй получить список доступных моделей из среды. Если
это не удалось, вопрос должен быть простым: «Пришлите список доступных моделей,
например строками или через запятую. Я сам предложу, какую использовать для
обычной работы, узкого поиска, анализа и проверки».

После получения списка моделей сам предложи понятную схему их использования:

- основная модель для обычной работы и финального ответа;
- быстрая модель для узкого поиска и извлечения фактов;
- модель для ограниченного анализа и сравнения;
- сильная модель для редкой независимой проверки ответственных решений.

Не требуй от пользователя объяснять цену, классифицировать модели или выбирать
внутренние назначения. Если данных о цене нет, пометь вывод как эвристику по
названию и возможностям, а не как проверенный прайсинг.

### Когда проверять политику

При корневой настройке проекта сначала прочитай проектный локальный слой и
определи состояние политики. Полную проверку запускай, если выполнено хотя бы
одно условие:

- пользователь явно попросил проверить политику;
- нет настроенной политики и нет состояния `creation_declined`;
- состояние — `configured`, а с `last_checked_at` прошло не меньше
  `interval_days`.

При отсутствующем `interval_days` предложи настроить срок; исходное предложение
— 30 дней. Нулевое или отрицательное значение, отсутствующую, нечитаемую либо
будущую дату считай ошибкой состояния и основанием для проверки. Если политика
свежая, назови следующую дату проверки и не запускай полный аудит.

Состояние `creation_declined` означает осознанный отказ пользователя и не
считается отсутствующей политикой. Оно не блокирует прямой запрос на проверку
или новое решение пользователя создать политику. Срок повторной проверки к
отказу не применяется: не придумывай для него следующую дату.

Используй в `subagents.local.toml` машинно читаемый блок:

```toml
[policy_review]
state = "configured" # либо "creation_declined"
interval_days = 30
last_checked_at = "2026-01-01"
```

Дату обновляй только после завершённой проверки, а не после попытки, которая
закончилась блокером. Проверка без записи возвращает предлагаемую дату в отчёте;
её сохранение остаётся отдельным подтверждаемым изменением. Если политика живёт
в пользовательском слое оснастки, этот блок всё равно остаётся проектным
состоянием и может храниться в исключённом из Git `subagents.local.toml`.

## Режим `policy`

Используй этот режим, если пользователь просит только черновик, сравнение,
оценку схемы, проверку рисков или явно запрещает правки.

Результат режима:

- цель оптимизации и нецели;
- классы задач и уровни моделей;
- контракты делегирования;
- правила эскалации;
- проверки качества;
- бюджетные ограничения;
- открытые решения и данные, которые нужно получить до внедрения.

Если пользователь просит перенести общую часть настройки в глобальную
конфигурацию, сначала покажи черновик переноса прямо в ответе: что переносится
глобально, что остаётся проектным переопределением, какие файлы будут изменены
после согласования и как будет проверен фактический запуск.

## Режим `implementation`

Короткие команды вроде «настрой подагентов», «настрой экономное использование
подагентов» и «настрой подагентов для этого проекта» означают `implementation`:
сначала коротко сформулировать политику маршрутизации, затем зафиксировать
рабочую конфигурацию в поддерживаемом слое проекта.

Не меняй файлы только если пользователь явно просит один черновик, сравнение
вариантов или проверку без внедрения. В задаче внедрения всё равно сначала
кратко перечисли состав политики в ответе, а затем укажи, где она
зафиксирована.

Если режим `implementation` требует конкретных моделей, а список моделей не
удалось получить автоматически и пользователь его ещё не дал, финальная
самопроверка обязательна:

- есть ли явный статус `ожидает список моделей`;
- нет ли требования классифицировать модели по назначениям;
- есть ли прямой вопрос к человеку о простом списке доступных моделей;
- является ли этот вопрос последним содержательным блоком ответа.

## Конфигурация Codex

Для Codex не считай настройку подагентов завершённой, если результат ограничен
`AGENTS.md`: нужен реальный конфигурационный слой, закреплённые модели и
проверяемый запуск.

Основной вариант для повторяемой схемы на одной машине — пользовательский слой
Codex: `$CODEX_HOME`, пользовательская конфигурация, профили и поддерживаемое
место для глобальных определений агентов в текущей версии Codex. Проектную
конфигурацию создавай или обновляй только для локальных переопределений:
особых моделей, более строгих запретов, команд проверки, правил приёмки,
проектных источников истины или временного слоя.

Не используй искусственные идентификаторы, если такого артефакта реально нет.
Для версии схемы сам выбери короткое имя по области действия и источнику
настройки. Покажи выбранное имя и версию в человекочитаемой записи:

```text
Схема подагентов: <имя>, версия <дата или версия>.
Источник: <источник>.
```

Если проект использует другую среду подагентов, сначала определи её
конфигурационный слой. Не выдавай политику маршрутизации за техническую
настройку.

## Локальное хранение и проверка моделей

### Оснастка, каталог и история

Определи оснастку в следующем порядке: явное имя пользователя, текущая среда и
встроенные средства, пользовательская и проектная настройки, найденные
исполняемые файлы. Несколько оснасток проверяй раздельно: одинаковое имя модели
не доказывает общую доступность.

Сначала вызови для каждой оснастки её нативный каталог через
`discover-subagent-models --adapter '<команда>'`. Контракт адаптера — JSON с
именем оснастки, моделями, скрытыми вариантами и поддерживаемыми усилиями.
Адаптер Codex делает `initialize`, уведомление `initialized` и постраничный
`model/list` с `includeHidden`. Семейство — например, «<семейство>» — служит только
условием выбора точной записи каталога. При ошибке верни «список не получен»;
тогда допустимы локальная настройка или простой вопрос пользователю, но не
перебор предполагаемых имён.

В уже используемом проекте рекомендуй `analyze-subagent-sessions` до выбора
ролей. Начальная настраиваемая эвристика достаточной истории: 10 интерактивных
родительских сессий, 30 завершённых ходов, два дня и сведения о моделях и
инструментах для большинства ходов. Анализатор выбирает точный `cwd`, исключает
дочернюю унаследованную историю из повторного подсчёта и не выводит текстов
запросов, содержимого сессий или внешних путей. Его результаты влияют на число
ролей, право записи и размер передаваемого контекста; при нехватке истории
используй структуру проекта и правила.

Настройки моделей и ролей по умолчанию принадлежат пользователю или локальной
рабочей копии, а не Git-репозиторию. Если у активной оснастки есть
пользовательский слой, используй его. В остальных случаях создай
`subagents.local.toml` из образца и добавь имя файла в `.git/info/exclude`.
Перед сохранением такой политики в Git отдельно спроси пользователя и назови
конкретный файл, который станет общим для команды.

Проектное состояние проверки и явный отказ также храни в
`subagents.local.toml`, даже если сами роли и модели находятся в пользовательском
слое оснастки. Отсутствие операционной политики при состоянии `configured`
считай повреждённым состоянием, которое требует новой проверки.

Проверяй каждую назначенную модель через `check-subagent-models`. Инструмент
вызывает адаптер по контракту `<команда адаптера> <модель>`: запрос идёт через
стандартный ввод, непустой ответ при коде выхода 0 означает доступность. Адаптер
может относиться к любой оснастке. Codex-адаптер — только пример.

Результат «недоступна» означает отказ модели или доступа. Результат «не
проверена» означает отсутствие адаптера, исполняемого файла, сети либо
тайм-аут. В обоих случаях не фиксируй новое назначение модели и запроси замену
или доступ. Короткая проверка модели не доказывает запуск подагента: его
проверяй отдельно по журналу и фактическому назначению модели.

## Выбор моделей

Когда список моделей известен, предложи схему сам:

- для обычной работы выбирай наиболее дешёвую или быструю модель, которая
  достаточно надёжна для координации, редактирования файлов и финального
  ответа в задачах обычного риска;
- для узкого поиска и извлечения выбирай самую быструю подходящую модель, если
  её результат можно проверить ссылками на файлы или источники;
- для ограниченного анализа выбирай модель среднего уровня или ту же модель,
  что для обычной работы, если отдельная средняя модель не даёт явной выгоды;
- для ограниченного независимого обзора небольшого набора файлов предпочитай
  экономичную модель проверки, если её качество можно принять родителем;
- для редкой независимой проверки выбирай самую сильную доступную модель или
  специализированную модель проверки, если она явно предназначена для проверки.

Если список содержит только одну или две модели, предложи минимальный вариант:
одна основная модель и, если возможно, отдельная модель для проверки.

## Право на запись

Не запрещай запись всем слабым моделям только из-за уровня модели. Запрещай
запись ролям, для которых она не нужна. Например, исследователь только для чтения
остаётся без записи, а отдельный patch-worker может выполнять малые обратимые
правки, если это дешевле прямого выполнения родителем.

Подагент с правом записи допустим только при всех условиях:

- родитель явно поручил правку и указал точные файлы или узкую область записи;
- правка малая, обратимая и не требует широкого проектного контекста;
- результат можно проверить различиями, форматированием, тестом или другой дешёвой
  проверкой;
- подагенту запрещено менять правила, навыки, шаблоны, `AGENTS.md`,
  архитектурные решения, требования и другие источники истины;
- подагенту запрещено принимать проектные решения, расширять область задачи и
  самостоятельно выбирать новые файлы для записи;
- родитель проверяет diff, запускает доступную проверку и сам принимает или
  отклоняет результат.

Если хотя бы одно условие не выполнено, оставляй запись родителю или более
сильной модели.

## Проверки качества

- Требуй структурированный результат для делегированных задач везде, где это
  возможно.
- Требуй ссылки на источники для фактических утверждений и ссылки на файлы для
  утверждений о коде.
- Требуй явную неопределённость при неполных данных.
- Отклоняй результат подагента, который незаметно расширяет область задачи.
- Для подагента с правом записи требуй список изменённых файлов, описание правки
  и проверку, которую должен выполнить родитель.
- Эскалируй, когда подагент сообщает о неуверенности, противоречивых данных,
  нехватке контекста, сбое инструмента или невозможности проверки.
- Не позволяй одной и той же дешёвой модели и подготовить, и финально принять
  результат с высоким влиянием.
