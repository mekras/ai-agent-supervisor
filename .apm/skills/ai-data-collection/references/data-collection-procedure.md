# Порядок сбора данных для развития продукта

## Назначение

Этот документ задаёт порядок сбора данных перед разработкой или изменением
правил, ролей, навыков, шаблонов, проверок и других содержательных элементов
продукта.

Он нужен для того, чтобы:

- не разрабатывать элементы продукта из предположений;
- не превращать локальные случаи в переносимые правила;
- не выдавать рабочие гипотезы за обоснованные решения;
- собирать материал в виде, пригодном для последующей проверки и использования.

Этот порядок применяется по решению полного режима `ai-work-control`, а не
автоматически перед каждым проектным изменением.

## Когда применять

Порядок сбора данных обязателен, если полный контроль или явный запрос
пользователя установил хотя бы один сигнал:

- не хватает проверяемых оснований для решения;
- источники или утверждения противоречат друг другу;
- основания нельзя проверить по репозиторию, доступным источникам или данным
  пользователя;
- область изменения, её адресат или границы ещё неизвестны;
- есть риск превратить единичный случай в переносимое правило, роль, навык,
  шаблон или проверку.

Наличие источника истины само по себе не требует отдельного сбора. Если полный
контроль уже зафиксировал достаточные и непротиворечивые основания, область и
ограничения, переходи к профильному навыку без повторного сбора.

## Что считается данными

Для этой процедуры данными считаются:

- существующие документы репозитория;
- действующие правила проекта;
- решения, уже зафиксированные в документах;
- запросы, замечания и подтверждения человека;
- наблюдаемые повторяющиеся проблемы в работе с ИИ;
- ограничения текущего состояния продукта;
- результаты предыдущих проверок применения ИИ.

Данными не считаются неподтверждённые предположения, удобные формулировки и
заранее выбранные решения.

## Цель сбора данных

Перед началом проектирования агент должен собрать достаточно материала, чтобы
ответить на вопросы:

- какую именно проблему применения ИИ нужно решить;
- где эта проблема наблюдается;
- повторяется ли она;
- можно ли решить её переносимым элементом продукта;
- какие ограничения уже действуют;
- чего ещё не хватает для решения.

Если на эти вопросы нельзя ответить, проектирование не должно переходить в
запись новых правил, ролей или навыков.

## Порядок сбора данных

### 1. Определить предмет сбора

Агент должен кратко зафиксировать:

- что именно предполагается разработать или изменить;
- зачем для этого нужен сбор данных;
- какое решение пока отсутствует;
- какие вопросы должны быть прояснены до проектирования.

На этом этапе нельзя подменять предмет сбора готовым решением.

### 2. Собрать существующий контекст из репозитория

Агент должен проверить:

- какие документы уже относятся к рассматриваемой теме;
- какие формулировки уже утверждены;
- какие ограничения уже действуют;
- нет ли в репозитории уже существующего решения для той же проблемы;
- какие части текущей документации могут стать источником данных для дальнейшей
  работы.

Если в репозитории уже есть частичное решение, оно должно быть зафиксировано как
исходный контекст, а не проигнорировано.

### 3. Собрать данные от человека

Если из репозитория нельзя получить достаточный контекст, агент должен собрать
данные от человека:

- какую проблему человек реально хочет решить;
- где она проявляется на практике;
- что уже пробовали делать;
- какой результат человек считает полезным;
- какие ограничения и ожидания нельзя нарушать.

Если ответов недостаточно, это фиксируется как нехватка данных, а не замещается
выводом агента.

### 4. Разделить собранный материал по типам

После первичного сбора агент должен явно разделить материал на:

- факты;
- наблюдения;
- допущения;
- выводы;
- предложения;
- открытые вопросы.

Без этого разделения нельзя переходить к разработке нового элемента продукта.

### 5. Выявить повторяемую потребность

Агент должен отдельно проверить:

- проблема единичная или повторяемая;
- относится ли она к применению ИИ, а не к частной инженерной задаче;
- нужен ли переносимый элемент продукта или достаточно локального решения;
- можно ли проверить пользу от будущего элемента.

Если повторяемая потребность не выявлена, добавление нового элемента в продукт
не обосновано.

### 6. Зафиксировать нехватку данных

Если в собранном материале остаются пробелы, агент должен явно записать:

- каких данных не хватает;
- почему без них нельзя уверенно проектировать решение;
- можно ли получить эти данные из репозитория;
- нужно ли дополнительное уточнение от человека;
- можно ли временно ограничиться аналитическим выводом без проектирования.

Нехватка данных должна фиксироваться явно, а не растворяться в расплывчатых
формулировках.

### 7. Подготовить материал к проектированию

Результат сбора данных должен быть достаточным для следующего шага, если из него
уже ясно:

- какая проблема подтверждена;
- чем она подтверждается;
- какие ограничения действуют;
- какие вопросы ещё остаются открытыми;
- можно ли переходить к разработке решения;
- какой именно элемент продукта имеет смысл проектировать.

Если это неясно, сбор данных должен считаться незавершённым.

## Минимальная форма фиксации результата

Итог сбора данных должен быть зафиксирован хотя бы в такой форме:

```text
Предмет сбора:

Предполагаемое изменение:

Что уже есть в репозитории:

Что известно от человека:

Факты:

Наблюдения:

Допущения:

Выводы:

Открытые вопросы:

Повторяемая потребность:

Недостающие данные:

Можно ли переходить к проектированию:
```

## Требования к качеству собранных данных

Сбор данных считается пригодным только если:

- материал отделён от проектного решения;
- факты не смешаны с допущениями;
- открытые вопросы не скрыты;
- видно, почему проблема относится именно к применению ИИ;
- видно, почему возможное решение должно быть переносимым;
- можно проверить, достаточно ли данных для следующего шага.

## Что нельзя делать при сборе данных

- подменять сбор данных готовой структурой решения;
- выводить новую роль или навык из одного частного случая;
- оформлять гипотезу как утверждённый факт;
- игнорировать уже существующие документы репозитория;
- скрывать нехватку данных за общими словами;
- переходить к проектированию только потому, что направление кажется разумным.

## Применение к продукту

При развитии продукта полный контроль фиксирует, достаточно ли уже известных
оснований для новых правил, ролей, навыков, шаблонов, проверок и документов.
Если он обнаружил один из сигналов этого порядка, проектирование останавливают
до сбора и приведения данных в проверяемый вид. При отсутствии сигнала сбор
данных не повторяют.
