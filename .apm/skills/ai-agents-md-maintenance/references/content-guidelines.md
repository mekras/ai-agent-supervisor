# Рекомендации по содержанию для AGENTS-навыков

- Опиши `WHAT/WHY/HOW` для каждого действия.
- Давай конкретные команды и их аргументы/флаги, где это влияет на результат.
- Избегай обзоров проекта и советов общего характера.
- Разделяй обязательное и рекомендуемое поведение.
- Если есть риски безопасности, явно задай порог и точку подтверждения человеком.

## Качества хорошего `AGENTS.md`

Хороший `AGENTS.md`:

- является машинной инструкцией, а не заменой `README.md`;
- содержит только правила, которые меняют поведение агента в задачах;
- объясняет `WHAT/WHY/HOW`: что делать, почему это важно и как выполнить;
- даёт конкретные команды, пути, ограничения, роли и форматы результата;
- избегает длинных обзоров дерева, истории проекта и общеизвестных советов;
- не хранит секреты, токены, временные гипотезы и непроверенные планы;
- задаёт ясную иерархию при нескольких файлах;
- не создаёт противоречий между корневым и вложенными инструкциями;
- остаётся достаточно коротким, чтобы не вытеснять рабочий контекст задачи;
- устойчив к повторному применению разными агентами и в разных сессиях.

## Что оставлять в машинной инструкции

Оставляй в `AGENTS.md` только операционные указания:

- обязательные команды перед сдачей результата;
- команды тестирования, форматирования, генерации и проверки;
- правила безопасности и Git-ограничения;
- проектные соглашения, которые агент не выведет из кода;
- порядок подключения локальных навыков, MCP-продуктов или контрольных слоёв;
- форматы ответа, если они реально влияют на приёмку;
- точные условия, когда правило применяется.

## Рекомендуемый каркас

У разных проектов `AGENTS.md` должен оставаться узнаваемым для человека. При
создании нового файла или содержательной переработке существующего файла
ориентируйся на такой порядок разделов:

1. Краткое правило языка, навыков или контрольных процедур, если оно влияет на
   каждый запуск агента.
2. Область проекта и главные принципы, которые агент не выведет из кода.
3. Проектные источники знания, документация или правила маршрутизации.
4. Команды сборки, проверки, форматирования и генерации.
5. `## Внесение изменений` — границы самостоятельных правок.
6. `## Git` — добавление файлов в индекс, commit, push и опасные операции.

Это каркас по умолчанию, а не требование к буквальным заголовкам. Не добавляй
пустые разделы и не переименовывай осознанно выбранные пользователем заголовки,
если текущая структура уже понятна и покрывает те же решения.

## Политика границы изменений и Git

При настройке проекта сначала сохрани существующую политику. Если её нет,
`ai-setup-project` должен предложить пользователю выбрать способ работы с
индексом Git. Навык сопровождения `AGENTS.md` записывает выбранный вариант, но
не выбирает его самостоятельно.

Общая граница изменений может быть записана так:

```markdown
## Внесение изменений

- Разрешается вносить изменения только внутри папки проекта.
```

Для индекса Git предложи отдельный выбор.

### Вариант 1. Все изменения

```markdown
## Git

- Все изменения, включая существовавшие до начала задачи, если они не исключены
  через `.gitignore`, сразу добавляй в индекс Git.
- Не делай commit и push без явной просьбы пользователя.
```

### Вариант 2. Только изменения агента

```markdown
## Git

- Добавляй в индекс Git только файлы и фрагменты, созданные или изменённые
  агентом в рамках текущей задачи.
- Сохраняй состояние индекса и рабочей копии, существовавшее на момент начала
  задачи. Если файл уже содержал изменения и собственные фрагменты нельзя
  надёжно отделить, не добавляй файл целиком в индекс и сообщи об этом.
- Не делай commit и push без явной просьбы пользователя.
```

Допустимы другие формулировки, если они сохраняют выбранное решение:

- граница самостоятельных правок ограничена папкой проекта;
- область автоматического добавления в индекс соответствует выбранному варианту;
- при варианте «только изменения агента» существовавшие изменения не попадают в
  индекс вместе с собственными фрагментами без надёжного разделения;
- commit и push выполняются только по явной просьбе пользователя.

Если пользователь уже изменил эти решения, не заменяй их другим вариантом
молча. Сохрани выбранную пользователем политику, если он не просил заменить
именно этот блок и если правило не конфликтует с более сильной инструкцией
текущей задачи.

Удаляй или выноси:

- длинные описания продукта;
- список всего дерева проекта;
- советы уровня «пиши качественно»;
- повторяющиеся запреты без рабочего способа действия;
- устаревшие команды и временные заметки;
- дубли `README.md`;
- правила, которые противоречат более специфичным файлам.

## Как формулировать правила

Пиши так, чтобы агент мог действовать без догадки:

- используй императивные, проверяемые формулировки;
- указывай конкретные команды и рабочие директории;
- задавай условия применения правила;
- объясняй причину там, где иначе правило кажется произвольным;
- отделяй обязательное от рекомендуемого;
- называй действия, требующие подтверждения человека;
- для рискованных операций указывай явную границу самостоятельности;
- не смешивай несколько обязанностей в одном расплывчатом абзаце.

Плохой признак: агент может выполнить правило несколькими несовместимыми
способами и все они будут выглядеть допустимыми.
