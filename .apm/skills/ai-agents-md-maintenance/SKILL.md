# Навык: сопровождение AGENTS.md

Используй этот навык для создания и изменения `AGENTS.md` как машинной точки
входа в проект. Цель — получить инструкции, которые агент сможет стабильно
применить в реальной задаче: понять область действия, выбрать правильные
команды, не нарушить границы проекта и не потратить лишний контекст.

## Когда применять

Применяй навык, если нужно:

- создать новый `AGENTS.md`;
- изменить существующий `AGENTS.md` после изменения структуры, команд,
  инструментов, процесса разработки или правил проекта;
- сократить перегруженный `AGENTS.md`;
- разделить один файл на корневой и вложенные инструкции;
- устранить конфликт между несколькими файлами инструкций;
- повысить стабильность применения инструкций после провала агента;
- подготовить содержательное изменение, которое затем должен проверить
  `ai-audit-agents-md`.

Не применяй навык для чистой проверки без редактирования. Для этого используй
`ai-audit-agents-md`.

## Границы

- Создание, изменение и сопровождение `AGENTS.md` остаются в одном навыке,
  потому что используют один рабочий цикл: определить цепочку инструкций,
  область действия, содержание, конфликты и проверку результата.
- Проверка остаётся отдельным навыком `ai-audit-agents-md`, потому что у неё
  другой выход: независимый отчёт о проблемах, а не изменение файлов.
- После существенного изменения `AGENTS.md` инициируй `ai-audit-agents-md` как
  контрольный этап.

## Обязательный порядок

1. Собери контур инструкций: все `AGENTS.md`, `AGENTS.override.md` и соседние
   файлы инструкций, которые могут участвовать в цепочке.
2. Определи задачу изменения: какое поведение агента должно измениться, какой
   риск закрывается и в каком файле должно жить правило.
3. Раздели уровни: корневой `AGENTS.md`, вложенные инструкции, профильные
   навыки, README и справочные документы.
4. Оставь в `AGENTS.md` только операционные указания: команды, ограничения,
   Git-правила, проектные соглашения, условия применения и форматы результата.
5. Сформулируй инструкции так, чтобы агент мог действовать без догадки:
   обязательность, условия, команды, пути, причины и точки подтверждения.
6. Стабилизируй иерархию: вложенные файлы уточняют область, а не молча отменяют
   корневой процесс.
7. Проверь стоимость контекста: удали повторы, общие советы, дубли README и
   длинные справки без рабочей пользы.
8. Если разделяешь инструкции между корнем и вложенным файлом, отдельно
   перечитай итоговое содержание: в корне не должны остаться локальные команды,
   пути, переменные окружения и дубли README, которые ты заявил как перенесённые
   или удалённые.
9. Проверь результат: ссылки, команды, отсутствие секретов, отсутствие
   конфликтов, сценарий применения и добавление новых файлов в Git по правилам
   проекта.

## Формат результата

Результат применения навыка должен содержать:

- изменённые или созданные файлы инструкций;
- краткое обоснование, какие проблемы закрывает изменение;
- список перенесённых или удалённых фрагментов, если были выносы;
- ограничения и недостающий контекст;
- проверку применимости результата;
- рекомендацию, нужна ли независимая проверка через `ai-audit-agents-md`.

## Ошибки применения

Ошибкой считается:

- создать большой `AGENTS.md` без реальной задачи применения;
- смешать документацию для человека и машинные инструкции;
- записать непроверенную гипотезу как действующее правило;
- добавить общие советы вместо конкретных команд и условий;
- расширить корневой файл локальными правилами подкаталога;
- убрать правило без понимания, какой риск оно закрывало;
- оставить конфликт между файлами инструкций;
- не проверить результат через `ai-audit-agents-md` после существенной правки.

## Связь с другими навыками

- `ai-audit-agents-md` — независимая проверка качества `AGENTS.md` после
  создания или изменения.
- `ai-work-control` — обязателен перед изменениями, которые становятся
  источником истины проекта.
- `ai-data-collection` — полезен, если нужно собрать факты о командах,
  структуре и реальных провалах до проектирования инструкции.
- `ai-work-result-evaluation` — нужен, если результат будет принят как
  основание для будущих изменений.
