# Настройка проекта под работу агентов

Перед существенным изменением источника истины проекта получи
`ai-work-control/full`; затем веди шаги без повторения контроля.

Корневая точка входа настройки проекта под правильную работу агентов. Навык не
делает предметную работу сам: он обследует проект, ведёт пользователя по шагам
настройки и делегирует каждый шаг профильному навыку коллекции. Свою процедуру
делегируемого навыка он не повторяет.

## Главное правило: ничего молча

Навык работает как последовательность шагов. Единица — осмысленная для
пользователя возможность, а не файл. Для каждого шага:

1. покажи конкретное изменение, которое предлагается внести;
2. объясни пользу для пользователя — что это ему даёт, а не какое правило или
   навык этого требует;
3. дождись отдельного подтверждения именно этого шага;
4. внеси изменение только после подтверждения;
5. не объединяй подтверждения разных шагов в один общий список.

При повторной настройке распознавай уже сделанное и не переспрашивай:
предлагай изменение, только если оно ещё не выполнено или разошлось с целью.
Проверку политики подагентов запускай в режиме без записи только в одном из
случаев:

- пользователь явно запросил проверку;
- в проекте нет ни политики подагентов, ни зафиксированного отказа от её
  создания;
- для существующей политики истёк срок со дня последней проверки.

При настройке спроси срок повторной проверки и предложи один месяц (30 дней)
как значение по умолчанию. Сохраняй срок, дату последней успешной проверки и
явный отказ в проектном локальном слое по процедуре `ai-setup-subagents`.
Отсутствующую или нечитаемую дату существующей политики считай просроченной.
Зафиксированный отказ подавляет автоматический запуск из-за отсутствия
политики, но не отменяет прямой запрос пользователя.

## Обязательный порядок

1. Определи корень и состояние проекта: какие настройки уже есть, какие навыки
   коллекции доступны, публикует ли проект коллекцию.
2. Проверь, нет ли в проекте уже установленного продуктового сценария настройки:
   явного правила в `AGENTS.md`, `CLAUDE.md` или соседних инструкциях, а также
   навыка из подключённой зависимости, который берёт на себя запросы вида
   «Настрой проект». Если такой сценарий есть, не продолжай настройку этим
   навыком: назови найденный маршрут и передай дальнейшую работу ему либо
   остановись до подтверждения человека.
3. Составь набор применимых шагов из перечня ниже и покажи план. Подтверждение
   запрашивай по каждому шагу отдельно в момент его выполнения, а не за весь план.
4. Выполняй шаги по одному, делегируя профильному навыку. Не дублируй процедуру
   делегируемого навыка внутри этого навыка.
5. На решениях, которые нельзя принять автоматически, останавливайся и передавай
   выбор человеку.
6. В отчёте назови выполненные шаги, делегированные навыки, что подтвердил
   пользователь и что осталось.

## Шаги настройки и делегирование

- Машинные инструкции проекта — агент стабильно следует правилам проекта.
  Базовым источником считай `AGENTS.md`, но учитывай соседние точки входа
  конкретной обвязки, например `CLAUDE.md`. Делегируй навыку сопровождения
  `AGENTS.md`; если проект использует Claude и отдельного `CLAUDE.md` нет,
  предложи создать `CLAUDE.md` с текстом `@AGENTS.md` или символьную ссылку
  `CLAUDE.md -> AGENTS.md`.
  Если в проекте ещё нет явной политики работы с индексом Git, предложи
  пользователю отдельным решением выбрать один из вариантов:
  1. сразу добавлять в индекс все неигнорируемые изменения, включая
     существовавшие до начала задачи;
  2. добавлять только изменения агента в текущей задаче, сохраняя состояние
     индекса и рабочей копии на момент начала задачи; файл с прежними
     изменениями не добавлять целиком, если собственные фрагменты нельзя
     надёжно отделить.
  Для каждого варианта объясни влияние на состояние Git. Уже зафиксированную
  политику сохрани без повторного выбора, если пользователь не просил её
  пересмотреть. После выбора делегируй запись правила навыку сопровождения
  `AGENTS.md`; не выбирай вариант за пользователя.
- Политика подагентов — предсказуемая и экономная маршрутизация подагентов.
  Если выполнено одно из условий периодической проверки, делегируй
  `ai-setup-subagents` проверку без записи. Исправления, фиксацию отказа и
  обновление даты успешной проверки предлагай отдельными подтверждаемыми
  шагами. Если ни одно условие не выполнено, не запускай полный аудит и назови
  дату следующей проверки для существующей политики. При зафиксированном отказе
  сообщи, что вернуться к вопросу можно по прямому запросу пользователя.
- Упаковка коллекции APM, если проект публикует коллекцию, — повторяемая
  установка, проверка и выпуск. Делегируй `ai-setup-apm`.
- Контроль правил через хуки, опционально, — автоматическая проверка правил на
  событиях. Делегируй `ai-setup-hooks`.

Контроль и приёмка работы ИИ — это не настройка файла, а правило применения.
Перед значимыми изменениями опирайся на навык контроля работы ИИ и аудит
проекта, но не превращай это в отдельный шаг-артефакт.

## Границы

- Навык не выполняет предметную работу делегируемых шагов сам.
- Чем коллекция не владеет — README, требования, предметные данные и доменная
  документация, — навык не делает. Перенаправляй такие задачи в соседние навыки
  или клиентские инструкции проекта, не опираясь на навыки других продуктов.
- Для проекта-потребителя APM не считай сухую проверку `apm compile`, которая
  показывает генерацию корневого `AGENTS.md`, достаточным основанием предлагать
  перенос рукописных правил в `.apm/instructions/**`. Такой перенос меняет
  источник истины проекта и требует отдельного маршрута сопровождения
  `AGENTS.md` и явного решения человека. Аналогично, наличие цели `claude` в
  `target` или `targets` не означает, что нужно дублировать правила руками:
  сначала предложи тонкую точку входа `CLAUDE.md`, ссылающуюся на `AGENTS.md`,
  а не отдельный расходящийся текст.
- Не вноси ни одного изменения без отдельного подтверждения шага.
- Останавливайся на решениях, которые должен принять человек.
