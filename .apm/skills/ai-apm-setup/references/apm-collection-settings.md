# Настройки коллекции APM

Эту справку читай перед созданием или изменением `apm.yml` для коллекции
навыков.

## Стоп-условие перед настройкой пакета

Сначала отдели публикуемый пакет APM от проекта, который только потребляет
пакеты APM как зависимости.

Само по себе наличие `apm.yml`, `apm.lock.yaml`, `dependencies.apm` или
`devDependencies.apm` не означает, что перед тобой коллекция APM. Эти признаки
могут относиться только к клиентскому подключению `ai-dev-team` или другого
пакета.

Ищи признаки публикации:

- `.apm/skills/**`;
- `.apm/agents/**`;
- публикуемые `.apm/instructions/**`;
- `type` или `includes` в `apm.yml`, которые описывают пакет проекта;
- `scripts.tests`, нацеленный на проверку публикуемой коллекции;
- явный запрос человека настроить именно пакет APM.

Если проект только потребляет `ai-dev-team` или другой пакет через
`dependencies.apm` или `devDependencies.apm`, не создавай структуру коллекции и
не переписывай манифест под producer-сценарий. В таком случае задачу нужно
вернуть в `ait-setup`, `ai-agents-md-maintenance` или клиентские инструкции
проекта.

## Минимальный манифест

Для коллекции навыков с несколькими примитивами исходное дерево APM должно
лежать в `.apm/skills/**`. Корневой `skills/**` допустим для сгенерированного
плагинового бандла или совместимой старой схемы, но не как основной источник
APM-пакета.

Для такой коллекции обычно нужны:

```yaml
name: project-name
version: 0.1.0
description: "Коллекция навыков"
targets:
  - codex
type: "skill"
includes: auto
dependencies:
  apm: []
  mcp: []
scripts:
  tests: "..."
```

Если проект уже указывает `target` или `targets`, сохрани это поле. В новом
манифесте предпочитай `targets` как список. Если целевая среда неясна, не
добавляй все возможные значения автоматически: возьми значение из проекта или
уточни у человека.

`includes: auto` нужен, чтобы локальное дерево `.apm/` разворачивалось при
установке пакета. Если `includes` уже задан явным списком, не перезаписывай его
автоматически, но проверь, что список включает нужные пути `.apm/skills/**`.

## Структура исходников

Нормальная структура пакета с несколькими навыками:

```text
project/
+-- apm.yml
+-- .apm/
    +-- skills/
        +-- example-skill/
            +-- SKILL.md
```

Если найден только корневой `skills/example-skill/SKILL.md`, зафиксируй
расхождение и перенеси исходники в `.apm/skills/example-skill/SKILL.md`, когда
задача разрешает правки. Не принимай успешный `apm pack` как доказательство, что
исходная структура APM настроена правильно: упаковка плагина может включать
корневой `skills/`, но `apm install` для локального пакета разворачивает
исходники из `.apm/` при наличии `includes`.

Если `apm pack --dry-run` показал один лишний файл, например `.pyc` или
`__pycache__`, это доказывает только локальный дефект публикуемого состава.
После его удаления всё равно проверь, не осталось ли структурных расхождений в
`.apm/skills/**`, `includes`, `scripts.tests` и различении между установкой и
упаковкой.

## Команда tests

`scripts.tests` должна запускать существующие проверки коллекции. Для
репозитория с валидаторами навыков используй:

```bash
sh -c 'set -e; target="${APM_EVAL_PATH:-.apm/skills}"; python3 tools/validate-skill-descriptions.py "$target"; python3 tools/validate-trigger-evals.py "$target" --require-all; python3 tools/validate-skill-result-evals.py "$target"; python3 tools/run-skill-evals.py "$target"'
```

Если таких инструментов нет, найди существующую команду тестов проекта. Если
тестов нет, сначала создай минимальную проверку коллекции или остановись и
сообщи, что `apm run tests` нельзя честно настроить без команды тестов.

## Выборочный запуск

Команда `scripts.tests` для коллекции навыков должна оставлять полный запуск
значением по умолчанию, но поддерживать узкий цикл разработки:

- `apm run tests` — полный набор коллекции;
- `APM_EVAL_PATH=.apm/skills/<name> apm run tests` — проверки одного навыка;
- `APM_EVAL_PATH=.apm/skills/<name> APM_EVAL_CASE_ID=<id> apm run tests` — один
  модельный сценарий.

Выборочный запуск не заменяет полный запуск перед публикацией, изменением общей
инфраструктуры тестов или правкой нескольких навыков. Он нужен, чтобы быстро
проверять локальную правку и не тратить модельные вызовы на незатронутые
каталоги навыков.

## Проверка после правки

Минимально проверь:

- `apm.yml` существует в корне проекта;
- исходники публикуемых навыков лежат в `.apm/skills/**`;
- `includes` отсутствует только при осознанной совместимой схеме; для обычной
  коллекции `.apm/skills/**` он задан как `auto` или явно включает нужные пути;
- `type` соответствует публикуемой коллекции;
- `scripts.tests` есть и ссылается на существующие файлы или команды;
- проверка бюджета `description` подключена, если коллекция публикует навыки;
- `apm run tests` запускается либо объяснено, почему запуск сейчас невозможен;
- для выпуска выполнены `apm view`, `apm audit` и `apm pack`, если они доступны
  и входят в задачу.

Если после правки менялся публикуемый состав, повтори `apm pack --dry-run` или
другую эквивалентную проверку упаковки. Не оставляй в отчёте только
доправочный результат упаковки.

Если используется `apm audit --ci --no-policy --no-drift`, зафиксируй, что
проверка дрейфа пропущена и покрытие снижено. Такой запуск полезен как
частичное подтверждение, но не заменяет структурную приёмку коллекции.

## Критерий завершения setup

Режим `setup` считается завершённым, когда агент может отдельно назвать:

- что подтверждено по манифесту;
- что подтверждено по исходному дереву `.apm/skills/**`;
- что подтверждено по `includes`;
- что подтверждено по `scripts.tests` и `apm run tests`;
- что подтверждено по публикуемому составу после внесённых правок;
- какие свойства не проверялись и какой остаточный риск остался.

Если агент может сослаться только на один исправленный артефакт, один запуск
`apm pack` или один запуск `apm audit`, режим `setup` завершать нельзя.

`apm compile` не обязателен для пакета, который публикует только
`.apm/skills/**`.
Он нужен, когда пакет содержит `instructions` или проект явно проверяет
компиляцию таких примитивов APM.
