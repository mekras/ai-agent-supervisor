# Архитектура коллекции

## Маршрутизация

Базовая логика маршрутизации:

1. Если задача явно меняет правила, роли, навыки, шаблоны, документацию или
   другой источник истины, основной маршрут — `ai-work-control`.
2. Если задача неясная, спорная или контекст неполный, сначала применяется
   `ai-application-check`.
3. Если перед проектированием нужно собрать и структурировать материалы,
   применяется `ai-data-collection`.
4. Если результат ИИ нужно принять, опубликовать или использовать как основание
   следующего шага, применяется `ai-work-result-evaluation`.
5. Если агент нарушил правило, пропустил обязательный шаг или неправильно
   применил маршрут, применяется `ai-rule-failure-analysis`.

Эта схема описывает архитектурную роль навыков. Операционные условия применения
зафиксированы в самих `SKILL.md`.

## APM

При подключении через `APM` коллекция должна работать через навыки. Поэтому
каждый навык обязан содержать собственные условия применения, порядок действий,
границы и ссылки на нужные материалы.

Для producer-стороны пакета это означает следующее:

1. Текущая коллекция опирается на skills как на основной переносимый primitive.
2. Если пакет будет расширяться, prompts, instructions и agents нужно
   добавлять осознанно по их роли, а не как взаимозаменяемые сущности.
3. Hooks и commands следует считать target-specific surfaces, а не основным
   кросс-харнесным способом доставки поведения.
4. `apm compile` нужен только для instruction-layer. Для skills-only пакета он
   не является обязательным шагом выпуска, но станет обязательной проверкой,
   если в репозитории появятся instructions для `claude`, `codex` и других
   harnesses, которые читают root context files.
